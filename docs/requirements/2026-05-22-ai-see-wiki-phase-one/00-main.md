## 1. 引言

### 1.1 目的

本文档描述 AI SEE Wiki 文档站第一期完整上线的软件需求，供站点维护者、内容作者、开发实现者、贡献者及后续 OpenSpec change 拆分使用。
各功能模块的详细 FR 见上方“模块文档索引”中的对应文档。

### 1.2 范围

AI SEE Wiki 是一个面向 AI Engineering 的开源双语文档站，核心聚焦 OpenSpec、Compound Engineering 和 aisee skill 生态。第一期交付范围包含可上线的 VuePress/Plume 文档站、P0/P1 中文完整正文、后续英文完整正文的双语体系、独立 skills 仓库的文档化集成、开源贡献与部署配套。

本文档不要求把 `aisee-*` skills 源码放入文档站仓库；skills 应独立维护，文档站提供 skill 介绍、仓库链接、安装命令和版本说明。具体文章正文内容在实现各内容 change 前逐篇深度讨论和确认，本文档定义范围、质量标准、流程和验收口径。

## 2. 整体描述

### 2.1 产品概述

AI SEE Wiki 不是普通 AI Coding 工具教程站，而是面向 AI 时代软件工程方法论的开放文档站。它通过“基础层 + OpenSpec 主线 + Compound/Harness 主线 + aisee skill 生态 + 综合工作流 + 原创思考 + 资源中心”的结构，帮助读者理解如何把需求、设计、实现、验证和复盘组织成可持续的工程系统。

当前仓库已存在 VuePress 2、vuepress-theme-plume、中文/英文 locale 基础配置和示例页面。第一期需求是在现有基础上完成正式站点信息架构、内容正文、双语体系、开源入口、部署说明和 quality gate 文档，使站点具备上线和后续持续扩展能力。

### 2.2 目标用户

| 用户角色 | 描述 | 主要诉求 |
|----------|------|----------|
| AI Engineering 学习者 | 希望系统理解 AI 编程工程化方法的个人开发者或团队成员。 | 通过清晰学习路径理解 OpenSpec、Harness、aisee 生态及其关系。 |
| OpenSpec / Compound / aisee 实践者 | 已经使用或准备使用相关工作流的工程师。 | 获取深入方法论、操作边界、模板、案例和工作流参考。 |
| 文档贡献者 | 通过 Issue/PR 参与文档修订、案例投稿或资源补充的人。 | 明确贡献范围、编辑入口、写作标准、许可证和构建检查方式。 |
| 站点维护者 | 负责内容规划、发布、部署、导航和质量检查的人。 | 管理中英双语内容、版本链接、部署配置、SEO/LLM 入口和独立 skills 仓库引用。 |
| aisee skill 使用者 | 希望了解、安装或升级 aisee skills 的用户。 | 查看每个 skill 的用途、适用边界、版本说明、安装命令和源码仓库链接。 |

### 2.3 运行环境

- 站点运行形态：静态文档站。
- 当前技术栈：VuePress `2.0.0-rc.30`、`vuepress-theme-plume` `1.0.0-rc.201`、Vue `3.5.34`、Vite bundler。
- 包管理约束：`package.json` 声明 `pnpm@10.33.2`，Node 版本要求为 `^20.19.0 || >=22.0.0`。
- 当前站点源目录：`docs/`，当前构建命令为 `pnpm docs:build`；第一期目标结构已确认为迁移到 `site/` 作为唯一站点发布源，并将协作产物保留在独立 `docs/`。
- 当前 locale：中文根路径 `/`，英文路径 `/en/`。
- 目标托管：GitHub public repository + Cloudflare Pages，生产域名为 `aisee.wiki`。

### 2.4 设计约束

- 第一阶段按“先中文完整正文，再英文完整正文”的顺序推进，但 SRS 范围包含中英双语完整交付。
- `aisee-*` skills 源码不放入文档站仓库，应使用独立仓库维护；文档站提供介绍、链接、安装命令和版本说明。
- 站点源码应按规划迁移到 `site/`，需求文档、UI 内容规格、技术上下文、复盘等协作产物保留在独立 `docs/`，避免发布源与协作产物混用。
- 文档正文应优先基于本地 Obsidian 参考目录和已确认规划文档改写，不应凭空编造工具行为或项目事实。
- SRS 不定义视觉设计、页面布局、组件样式或具体实现代码；这些内容由后续 UI 内容规格、设计或实现 change 处理。
- 用户已确认本阶段不要求截图验收；构建和本地预览可由维护者自行执行。

## 4. 非功能需求

### 4.1 性能

| 指标 | 要求 | 备注 |
|------|------|------|
| 静态站点构建 | 站点应能通过项目声明的构建命令完成静态产物生成。 | 本阶段可不由代理执行，但需提供可运行命令。 |
| 页面导航体验 | 首页、栏目首页、文章页和资源页应保持可发现、可浏览。 | 不要求截图验收。 |
| 搜索 | 站点应启用或保留本地搜索能力。 | 当前 Plume 配置已启用 local search。 |

### 4.2 安全

- 仓库不得提交私密 token、cookie、私钥、私有证书、生产凭据或个人敏感数据。
- 文档站公开内容不得直接发布 `docs/` 协作草稿、未清理的本地路径、内部私人笔记或未确认授权的外部内容。
- skills 独立仓库链接、安装命令和版本说明必须指向公开、可审查的来源。
- 若引用外部项目、工具或文档，应标注其非官方/独立实践性质，避免误导为官方文档。

### 4.3 可靠性

- 文档站应保持稳定的信息架构，避免频繁变更已发布 URL。
- 中英文同主题页面应使用一致 slug，便于维护 `hreflang`、canonical 和跨语言跳转。
- skills 版本说明应能反映文档所描述的 skill 行为，降低文档与独立仓库漂移风险。
- 开源贡献、部署和构建说明应足以支持维护者在本地复现站点预览。

### 4.4 可维护性

- 栏目、导航、集合、资源页和文章路径应按规划的内容边界组织，避免把草稿产物、skill 源码和公开站点内容混放。
- 每篇文章应遵守统一写作结构：解决问题、核心结论、背景动机、工作机制、最小示例、常见误区、适用边界、与 OpenSpec/Compound/aisee 的关系、下一步阅读。
- 内容事实源应可追溯到规划文档、本地参考资料或明确标注的外部资料。
- 开源配套文件应说明贡献流程、许可证、构建命令、编辑入口和内容质量要求。

### 4.5 其他

- 国际化：中文为默认语言，英文路径为 `/en/`；第一阶段执行顺序为中文优先，再英文完整补齐。
- SEO：站点应配置基础标题、描述、站点域名、站点地图、编辑链接、必要的中英文元信息。
- LLM 友好：站点应提供或规划 LLMs.txt 能力，便于 AI 工具引用站点内容。
- 无障碍：文档内容应使用语义化标题层级、可读链接文本和可检索的 Markdown 内容。
- 开源：文档采用 `CC BY-SA 4.0`，代码示例采用 `MIT`。
